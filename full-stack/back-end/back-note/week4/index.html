# HashSet 集合

HashSet:

特点：无序 唯一

数据结构：哈希表

基本用法及特点：

基本用法及特点

创建对象：HashSet <泛型> set = new HashSet<>();

添加元素：set.add (元素)/Collections.addAll (集合，元素，元素…)

集合大小：set.size ()

删除元素：set.remove (元素)

判断包含：set.contains (元素)

遍历：foreach + 迭代器

集合并集：集合 A.addAll (集合 B)

集合差集：集合 A.removeAll (集合 B)

集合交集：集合 A.retainAll (集合 B)

HashSet 唯一性

唯一：内存里面的同一个对象 无论方法怎么覆盖 都不可以添加多次

“唯一”：内存里面的两个对象 被视为相等对象 从而只能添加一次

HashSet 的唯一性是靠覆盖 hashCode ()、== 和 equals () 实现的

HashSet 添加元素的时候完整比较流程：

hashCode () ==[地址] equals ()

–HashSet 底层的 add (元素)、remove (元素) contains (元素) 底层尊重 hashCode () == equals () 比较机制

HashSet 底层的 add (元素)/remove (元素)/contains (元素) 底层尊重 hashCode () == equals ()

并且修改删除是先在哈希表里找，然后修改删除，所以如果修改的属性关乎哈希值，要一删，二改三添加

HashSet 构造方法其实可以穿两个参数

第一个参数：int 分组组数 程序员可以随意的制定分组组数

​ 但是最终一定会变成 2 的 n 次方数

第二个参数：float 加载因子

​ 阈值：达到某个条件的最小临界值

​ 哈希表想要扩容的最小临界值

​ 阈值：分组组数 * 加载因子

​ 如：当班级温度至少为 20 度的时候才能开空调

​ HashSet set = new HashSet();

​ 默认开辟 16 个小组 加载因子 0.75F 阈值：12

–如果某组达到了阈值，下一个加入的元素如果不是加到空组里，就开辟

# SortedSet: 接口

TreeSet:

特点：有序 唯一

数据结构：二叉树

TreeSet1: 基本用法及特点

创建对象：TreeSet <泛型> set = new TreeSet<>();

添加元素：set.add (元素)、Collections.addAll (set, 元素，元素…)

集合大小：set.size ()

判断是否包含：set.contains (元素)

删除元素：set.remove (元素)

foreach + 迭代器

得到集合里面的第一个元素：set.first ()

得到集合里面的第一个元素并且删除：set.pollFirst ()

得到集合里面的最后一个元素：set.last ()

得到集合里面最后一个元素并且删除：set.pollLast ()

TreeSet2: 唯一性

唯一：内存里面的同一个对象 不能往 TreeSet 添加多次

“唯一”：内存里面的两个对象 视为相等对象 只能往 TreeSet 集合添加一次

TreeSet3: 验证唯一性

任何一个引用数据类型 如果想要装进 TreeSet 集合的话 需要当前类型会排序

当前的类型 implements Comparable <类型> -》 compareTo (参数)

Comparable 比较能力【接口】compare 比较

<泛型里放比较的类型>【关乎 compareTo 的参数类型】

实现这个接口说明这个类有比较能力

TreeSet4: 如果想要通过多个属性综合排序的话 优先比较什么属性 就描述如果这个属性不一样的话…

TreeSet5: 当我们在覆盖 compareTo () 的时候 尽量保证这个方法有机会返回 0 否则：

add (元素)：不能确保唯一性

remove (元素)：永远删除失败

contains (元素)：永远返回 false

如果 compareTo () 没有机会返回 0 的话 想要删除：car.remove ()

TreeSet6: 当我们使用迭代器在遍历集合的时候 不允许对集合的整体进行添加 / 删除 否则触发 CME 异常 如果在遍历的时候想要删除：car.remove ()

TreeSet7: 当一个对象已经装进 TreeSet 集合之后 不要随意的修改那些参与排序的属性值 如果在添加

之后想要修改的话：删除 + 修改 + 添加

TreeSet8: 第一个添加进 TreeSet 集合的元素在某个时间段内跟节点 但是随着元素的增多 根节点可能

发生改变

TreeSet9: 如果想要脱离开一个类制定他的比较规则

比较器： Comparator -> compare ()

一般为单例

list.forEach (System.out::println); // 依次遍历输出

list.forEach ((x)->System.out.println (x.name+“此次考试”+x.score+“分”)); // 依次遍历做 x

//stream ()：将集合流式化

//filter ()：过滤筛选

list.stream ().filter () 返回一个集合

# Map 键值对类型的集合

每次往集合添加 key [唯一]=value

所有 Map 集合通用的方法：

创建对象：HashMap <主键，值> map = new HashMap<>();

添加元素：map.add (元素)

集合大小：map.size ()

通过主键得到值对象：map.get (主键)

删除一整条记录：map.remove (主键)

判断某个主键是否存在：map.containsKey (主键)

判断某个值对象是否存在：map.containsValue (值)

map 的遍历方法：

通过 map 集合得到所有得主键：

Set <主键> set = map.keySet ();

遍历 set 得到每个主键：x

通过主键得到值对象：map.get (x)

获得主键，相当于主键分支，可以通过修改 set 集合修改 map

通过 map 集合得到所有的值对象：

Collection <值> cs = map.values ();

遍历集合得到所有的值：x

获得值，相当于主键分支，可以通过修改 Collection 集合修改 map

通过 map 集合得到所有的记录 [主键 + 值]

Set<Map.Entry < 主键，值>> set = map.entrySet ();

遍历集合得到每条记录：x

得到主键对象：x.getKey ()

得到值对象：x.getValue ()

修改值对象：x.setValue (值)

map 集合一旦主键确定，再添加覆盖主键就不会被修改，只会覆盖修改值

HashMap

HashMap 底层基于哈希表实现的 默认分 16 个小组 每个对象应该去到哪一个小组

根据哈希码值决定 -》 hashCode ()

每对记录 [主键 + 值] 应该去到哪一个小组 根据主键的哈希码值决定

当来到某个小组之后 为了保证主键的唯一性 需要拿着新来的主键和老元素的主键做 hashCode () == equals () 比较

HashMap 底层的：

put (主键，值)

get (主键)

remove (主键)

containsKey (主键)

底层尊重 hashCode () == equals () 比较机制

TreeMap

TreeMap 底层基于二叉树实现的 每对元素应该放在左子树还是右子树 根据新来元素的主键和集合里

面老元素的主键做 compareTo ()/compare () 比较

如果结果 < 0 新来的一对元素放在左边

如果结果 = 0 新来的一对元素舍弃【主键不变 值替换】

如果结果 > 0 新来的一对元素放在右边

TreeMap 底层的：

put (主键，值)

get (主键)

remove (主键)

containsKey (主键)

size () 貌似也是

底层尊重 compareTo ()/compare () 比较机制

# 面试

>HashMap 和 Hashtable 之间的区别？

同步特性不同

Hashtable 同一时间允许一个线程进行访问 效率较低 但是不会出现并发错误

HashMap 同一时间允许多个线程进行访问 效率较高 但是可能会出现并发错误

jdk5.0 开始 集合的工具类里面提供一个方法 将线程不安全的 HashMap 变成线程安全 Map 集合 所以 Hashtable 渐渐被淘汰了

Map map = ollections.synchronizedMap (HashMap 对象);

对 null 的处理不同

Hashtable 无论是主键还是值对象 都不能传 null 否则触发空指针异常

HashMap 无论是主键还是值对象都可以传 null 但是由于主键是唯一的 所以主键只能传一个 null 但是值对象可以穿多个 null

分组组数不同

Hashtable 默认分 11 个小组 程序员可以随意的指定

HashMap 默认分 16 个小组 程序员可以随意的指定 但是最终一定会变成 2 的 n 次方数

出现的版本不同

Hashtable since jdk1.0

HashMap since jdk1.2

>HashMap Hashtable ConcurrentHashMap 之间的区别？

Hashtable 为了追求安全性 当某个线程进入哈希表某个小组处理元素的时候 会将整个哈希表加锁 所以同一个时间最多只能一个线程操作哈希表

HashMap 为了追求高效性 将不再加锁 可能会出现的问题是一个线程在处理某个小组的时候又来一个线程同事处理该小组 可能会引发并发问题

ConcurrentHashMap 安全性和高效性都介于 Hashtable 和 HashMap 之间

采用了锁分离的机制 当某个线程处理某个小组的时候 只是将该小组加锁 如果又来一个线程想要处理该小组 需要等待 如果想要处理其他的小组 可以直接处理

>HashMap 在 jdk7.0 之前和 jdk7.0 之后的变化

jdk7.0 之前 哈希表底层基于数组 + 链表组成的

​ 数组：用来装表头信息 为了方便元素更加快速的找到应该去到哪一个小组

​ 组内采用是链表数据结构进行添加 / 删除元素

​ 7.0 之前 采用头插法

jdk7.0 开始 哈希表底层数组 + 链表 + 二叉树实现的

​ 组内采用的是链表和二叉树实现

​ 如果组内元素 <= 7 采用链表实现

​ 如果组内元素 > 7 个元素 将链表改成二叉树 [为了效率高]

​ jdk7.0 开始 采用尾插法
