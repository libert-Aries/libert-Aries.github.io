#: 运行时异常和非运行时异常的区别？

非运行时异常在编译的时候就要求程序员必须给出处理方案

​ 否则编译无法通过

​ 它们都直接继承自 Exception

运行时异常在编译的时候不要求给出处理方案 可以直接编译通过

​ 问题会在运行的时候直接体现出来～

​ 它们继承自 RuntimeException

时至今日 我们所见到过的运行时异常有哪些？

ArithmeticException => 算术异常

​ *：除以 0 了 找代码当中 / 它后面的变量可能是 0

NegativeArraySizeException => 负数的数组大小异常

​ *：初始化数组的时候 [] 当中要求指定元素个数

​ 而如果指定的是个负数 将直接导致该异常

ArrayIndexOutOfBoundsException => 数组索引值超出边界异常

​ *：访问数组元素的时候 我们会在 [] 当中写下标索引值

​ 可用范围是：0 - 数组元素个数 - 1

​ 找 [] 当中的数值 是否在可用范围

NullPointerException => 空指针异常

​ *：Java 当中引用类型的默认值是 null

​ 它代表对象根本不存在

​ 用它访问任何属性 调用任何方法都会导致空指针

StringIndexOutOfBoundsException => 字符串索引值超出边界异常

​ *：String 类提供了多个使用下标索引值操作字符串内容的方法

​ 例如:charAt () /substring ()

​ 如果传入的下标索引值不在可用范围 将会导致该异常

NumberFormatException => 数字格式异常

​ *：大多数包装类类型都提供了从字符串到对应基本数据类型转换的方法

​ 例如 Integer.parseInt () / Double.parseDouble () …

​ 如果传入的字符串有非法内容 则导致该异常

ClassCastException => 类型造型异常

​ *：强制类型转换的时候 要转换的对象跟目标类型不是一个分支

​ 则导致该异常

IllegalArgumentException => 非法参数异常

​ *：初始化 ArrayList 的时候 构造方法的参数

​ 可以指定底层 Object 数组的原始大小

​ 如果指定的是负数 将直接导致该异常

IndexOutOfBoundsException => 索引值超出边界异常 (集合)

​ *：List 作为有序集合 是可以通过下标直接操作的

​ 例如：get (int) /remove (int)

​ 如果传入的下标值非法 将导致该异常

IllegalStateException => 非法状态异常 (迭代器)

​ *：当迭代器光标没有指向元素的时候

​ 就直接调用了 car.remove ()

​ 将导致该异常～

ConcurrentModificationException => 并发修改异常～

​ *：此处省略一堆字

为什么要处理异常：

A. 非运行时异常 如果不做处理 编译都无法通过

B. 一旦程序运行过程当中出现异常而未作处理

​ 虚拟机会直接中断程序的执行～

如何处理异常：

1. 抛还上级 throws

​ 足以解决 A 但无力解决 B

​ 方法签名上有 throws 不是代表必然出异常

​ 它代表本方法执行过程当中可能出现指定的异常

​ 只是出现之后本方法不做处理 抛还给调用的上级处理

【抛到虚拟机上后若出现异常就中止，但未出现虚拟机就放行】

【若出现异常，虚拟机中止后，后面的代码就不会再运行了】

2. 自行处理 try catch finally

​ try{

​ 可能出现异常的语句

​ 通常只有一句

​ 除非需求决定

​ 前者出现异常 后者需要跳过…【虚拟机也直接叫停，但只是这一个 {} 里】

​ } catch (要捕获的异常类型 异常代号){

​ 对捕获到的异常进行处理：

​ 0. 隐瞒不报

​ 1. 简要的审 System.out.println (e.getMessage ());

​ 2. 详细的审 e.printStackTrace ();

​ 3. 谎报

​ }finally{

​ 无论是否出现异常 最终都要执行的操作

​ 通常 会是释放和关闭资源的操作～

​ }

​ *: 一个 try 后面可以跟上多个 catch 分支

​ 但是多个 catch 必须按照从小到大或者并列的顺序

​ 也必须将 catch 分支写多个

​ 从 JDK7.0 开始 可以使用多重 catch

​ 多种异常的类型之间 用 | 隔开

​ | 前后也需要满足前者不能包含后者这一特性

​ *: finally 当中永远不该出现 return 语句

​ 否则 try catch 当中的 return 就不可能执行了

​ 不一定！System.exit (0); 【叫停虚拟机】

什么是锁标记？

为了解决操作不完整性问题 Java 当中每个对象都配有一个锁标记

目的是让 线程 拿到这个锁标记才能进行完整的一组操作

而线程由于未拿到锁标记 则不能进入 {} 当中执行操作

同时进入对象的锁池当中产生阻塞 直到可以获得锁标记为止

什么是锁池？

Java 当中每个对象都有一份的空间 用于存放线程的

用于存放那些想要申请对象的锁标记 但是还没有拿到的线程对象的空间

什么是时间片？

CPU 允许线程任务执行一次的时间单位 被形象成称之为 "时间片"

什么是死锁 (DeadLock)?

互斥锁标记使用过多或者使用不当

就会造成多个线程相互持有对方想要申请的资源

不释放的情况下 又去申请对方已经持有的资源

从而双双进入对方已经持有的资源的锁池当中

产生了永久的阻塞…

如何解决死锁

涉及到一块空间 和 三个方法

一块空间： 对象的 等待池

三个方法： wait () /notify () /notifyAll ()

​ wait () : 告诉当前线程 放弃已经持有的锁标记

​ 并且进入调用方法的那个对象的等待池当中

​ notify () : 从调用方法的那个对象的等待池当中

​ 随机的唤醒一个线程

​ notifyAll () : 从调用方法的那个对象的等待池当中

​ 唤醒所有阻塞中的线程～

​ *： 注意 这三个方法不是 Thread 类的方法

​ 而是 Object 类的方法

​ 因为每个对象都有等待池

​ 每个对象都需要操作等待池

​ *： 注意 这三个方法都必须在已经持有

​ 对方锁标记的前提下才能使用

​ 否则不但操作失败 还会触发运行时异常

​ IllegalMonitorStateException

​ 在没有持有锁标记的情况下 直接操作对象的等待池！

从而在用户出现的第一时间 直接满足用户对资源的需求

并且将资源的创建和销毁都委托给资源池完成

从而优化用户体验～

IBM 的程序员曾经质疑官方对于线程的实现

假如一个线程的完整执行时间为 T

​ T = t1 + t2 + t3

​ t1 : 在操作系统当中映射一个线程的时间

​ t2 : 执行线程核心任务的时间 run ()

​ t3 : 在操作系统当中销毁一个线程的时间

假设 run () 当中代码非常简短 则 t2 所占 T 的比例就会很小

我们就会感觉付出和回报不成比例 喧宾夺主 主次不分…

从 JDK5.0 开始 由 DougLea 在 JUC 并发包当中提供的线程池的实现～

*: 核心类库当中由官方提供的常用的线程池种类有哪些？

1> 修复后可重用的 (预留多少活跃线程)

​ newFixedThreadPool(int)

2> 缓存机制的

​ newCachedThreadPool()

3> 单一实例的

​ newSingleThreadExecutor()

*: 阿里不允许直接使用官方提供的线程池实现

要求程序员自己创建线程池执行器

那么自己创建线程池执行器 需要五个重要参数 它们是？

​ 1> 线程池当中核心线程数量

​ 2> 线程池最大线程数量

​ 3> 保持活着的时间 KeepAliveTime

​ 4> 时间单位 TimeUnit

​ 5> 一个集合 一个队列 用于存放排队的线程任务

》ArrayList & Vector 哪个支持多线程并发操作～

ArrayList

​ 底层可能会出现多个线程同时操作集合

​ 所以效率较高 但是可能出现并发错误

​ *：正因为可能出现错误 我们说它不支持并发操作！

Vector

​ 底层不会发生多个线程同时操作集合

​ 当真正有多个线程尝试一起操作的时候

​ 底层的加锁机制会让它一个一个来

​ 所以它不会出现并发错误

​ *：所以我们说它支持并发操作

》Error 和 Exception 有什么区别？

Error 通常是指由于底层硬件环境或者系统原因导致的

​ 程序员通过编码无法解决的

​ 相对较严重的问题

Exception 就是程序运行过程当中出现的例外情况而已

​ 相对较轻

》运行时异常和非运行时异常的区别？

非运行时异常在编译的时候就要求程序员必须给出处理方案

​ 否则编译无法通过 【day23-TestException】

​ 【day23-TestPlus】

​ 它们都直接继承自 Exception

运行时异常在编译的时候不要求给出处理方案 可以直接编译通过

​ 问题会在运行的时候直接体现出来～

​ 它们继承自 RuntimeException
